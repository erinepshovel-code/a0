# GPT-5.5 Awareness Audit + Slug Bump

Date: 2026-04-25
Scope: OpenAI provider only. Non-OpenAI providers (Gemini, Claude, Grok)
were intentionally not touched.

## What changed

### `python/config/openai_policy.json`
- `version` bumped: `2026-04-06.a0.openai.policy.v1` →
  `2026-04-25.a0.openai.policy.v2`.
- Role slugs bumped to the matching `gpt-5.5*` snapshot:
  - `root_orchestrator`: `gpt-5.4` → `gpt-5.5`
  - `high_risk_gate`:    `gpt-5.4` → `gpt-5.5`
  - `worker`:            `gpt-5.4-mini` → `gpt-5.5-mini`
  - `classifier`:        `gpt-5.4-nano` → `gpt-5.5-nano`
  - `deep_pass`:         `gpt-5.4-pro` → `gpt-5.5-pro`
- `defaults.reasoning.effort`: `low` → `medium` (per the documented 5.5
  default; 5.4-era files still pinned `low`).

### Per-role `reasoning.effort` (deliberately preserved as-is)
The role-level overrides are explicit choices, not inherited 5.4 defaults,
so they survive the bump verbatim:

| role              | effort   | rationale                                |
|-------------------|----------|------------------------------------------|
| root_orchestrator | low      | cheap routing/synth; explicit            |
| high_risk_gate    | medium   | matches new default; explicit            |
| worker            | none     | batch passes; explicit                   |
| classifier        | none     | strict-JSON triage; explicit             |
| deep_pass         | high     | think-hard pass; explicit, gated         |

If you want root_orchestrator to ride the new `medium` default, drop the
`reasoning.effort` key from that role; otherwise leave it as is.

### `python/services/openai_router.py`
- The code-level fallback in `resolve_role_config` (used only if the policy
  file is missing `defaults.reasoning.effort` entirely) bumped from `"low"`
  to `"medium"` so the fallback also reflects the 5.5 era.
- **`resolve_model` precedence change (important).** Previously this
  function *required* the `OPENAI_MODEL_ROOT/WORKER/CLASSIFIER/DEEP` env
  vars and would raise if any were unset, which meant the `model` field
  in `openai_policy.json` was effectively documentation only — a slug
  bump in the policy file silently did nothing at runtime. Now the env
  var still wins (operator hot-pin), but if it's unset we fall back to
  the role's `model` field in the policy file. This makes the GPT-5.5
  bump in the policy actually take effect when env vars are absent or
  haven't been updated. Operators can still pin a specific snapshot via
  env var without editing the policy.

### `python/services/inference.py`
- `_call_openai_responses(reasoning_effort=...)` function-default bumped
  from `"low"` to `"medium"`. Callers in the routed path always pass
  `call_cfg["reasoning_effort"]` explicitly, so this is belt-and-suspenders
  for any future direct caller.

### `python/services/energy_registry.py`
- GPT slot pins (`gpt-5`, `gpt-5-mini`, `gpt-5-nano`) intentionally **left
  unversioned**. Added a one-line comment recording the intent: this surface
  is the user-facing optimizer presets, where OpenAI's latest-alias rolls us
  forward through the 5.x series without a config edit. Deterministic
  version pinning lives in `openai_policy.json` for the routed-call path.

## What did not change (and why)

- **Responses API path**: already routed end-to-end through
  `_call_openai_routed` → `openai_router.make_call_config` →
  `_call_openai_responses` against `_OPENAI_RESPONSES_URL`
  (`https://api.openai.com/v1/responses`). Nothing structural to migrate.
- **CoT / "step by step" prompt nudges**: grep across `python/` (services,
  routes, agents, config) found **zero** "step by step", "think step",
  "show your work", or "chain of thought" nudges in our own prompt
  surfaces. The only hits were inside user-uploaded `uploads/*.json`
  conversation imports, which are user data, not prompts we send. Nothing
  to remove.
- **`phase` parameter preservation**: we do **not** replay assistant items
  by hand on the OpenAI path. The Responses-API loop in
  `_call_openai_responses` only re-feeds `function_call` items and
  `function_call_output` results into the next round (lines 720-723);
  message/reasoning items are deliberately excluded to avoid 4xx errors.
  Because we never reconstruct an assistant `message` item ourselves, the
  `phase` parameter never enters our code path and no preservation work is
  needed. This assumption is recorded here so future work that *does* start
  replaying assistant messages knows to forward `phase` through.
- **Other providers** (Gemini / Claude / Grok / aimmh-lib): out of scope.
- **Per-role token budgets**: out of scope; only slug + effort defaults +
  prompt-string nudges were in scope.

## Operator notes — env vars

`OPENAI_MODEL_ROOT`, `OPENAI_MODEL_WORKER`, `OPENAI_MODEL_CLASSIFIER`,
and `OPENAI_MODEL_DEEP` (consumed in `openai_router._MODEL_ENV_MAP`) now
**override** the policy file's `model` field per role. With the new
precedence:

- env var set → that snapshot is used (operator hot-pin)
- env var unset → the policy file's `roles.<role>.model` is used

If you previously had these env vars pinned to `gpt-5.4*` snapshots,
update them to the matching `gpt-5.5*` slugs (or unset them entirely to
let the policy file drive). Otherwise the env-var pins will continue to
serve 5.4 traffic regardless of the policy bump.
